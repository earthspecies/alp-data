# RFC: Dataset Schema System

**Status:** Draft
**Author:** Gagan
**Date:** 2026-03-03
**Branch:** `gagan/schema-core`

## Context

We have a schema system in `esp_data/schema.py` that defines `ColumnSchema` and `DatasetSchema` Pydantic models. The `Dataset` base class supports an optional `schema` attribute that is validated after `_load()`.

**Current state:**
- Only **AnimalSpeak** defines a schema ‚Äî and it only declares 4 required columns out of 46 actual columns
- The abstract `DType` system maps to polars/pandas types: `str`, `int`, `float`, `bool`, `datetime`, `list[str]`, `list[int]`, `list[float]`, `nullable`

## Question 1: What is the schema for?

The schema system could serve different purposes, and the answer shapes the design:

| Purpose | Implication |
|---------|-------------|
| **Validation** ‚Äî catch data breakage when underlying files change | Schemas should list all columns with strict dtype checks |
| **Documentation** ‚Äî make datasets discoverable and self-describing | Schemas should be exhaustive, possibly auto-generated |
| **Contract** ‚Äî define the columns downstream consumers can rely on | Schemas should list only "important" columns, the rest are unstable |

**Proposed answer:** Start with **contract** ‚Äî schemas declare the stable interface of a dataset. Columns not in the schema exist but are not guaranteed across versions. Validation enforces the contract at load time.

**Alternative:** Exhaustive schemas that document everything. Risk: high maintenance burden and frequent breakage when upstream data changes.

> **Author's position:** Contract. The hand-written schema defines the stable interface ‚Äî the columns downstream consumers can depend on. But discoverability also matters (see Q2), so we pair this with a separate auto-generated exhaustive schema for search/versioning tools.

> **Discuss:** Which purpose resonates most? Should we support both a "contract" schema and a "full" schema?

## Question 2: Exhaustive vs. minimal schemas

AnimalSpeak has 46 columns but its schema lists 4. Two approaches:

### Option A: Contract columns only

```python
schema = DatasetSchema(
    columns=[
        ColumnSchema(name="local_path", dtype="str", required=True),
        ColumnSchema(name="species_scientific", dtype="str", required=True),
        ColumnSchema(name="species_common", dtype="str", required=True),
        ColumnSchema(name="caption", dtype="str", required=True),
    ]
)
```

- Validation only checks these 4 columns exist with correct dtypes
- Extra columns are allowed and ignored
- Low maintenance, but consumers don't know what else is available

### Option B: Full schema with required/optional distinction

```python
schema = DatasetSchema(
    columns=[
        ColumnSchema(name="local_path", dtype="str", required=True),
        ColumnSchema(name="species_scientific", dtype="str", required=True),
        ColumnSchema(name="caption", dtype="str", required=True),
        # ... all 46 columns, most with required=False
    ]
)
```

- Validation checks required columns strictly, optional columns only validated if present
- More documentation value, but heavy to maintain

### Option C: Hybrid ‚Äî contract schema + auto-generated full schema

- Hand-written `schema` on each class lists contract columns
- `extract_schemas.py` generates a separate reference doc with full column listings
- Best of both worlds, but two sources of truth

> **Author's position:** Option C (hybrid), with a clear separation of concerns:
> - **Contract schema** = hand-written on the dataset class, used for load-time validation. Small, stable, maintained by hand.
> - **Exhaustive schema** = auto-generated by CI as **structured JSON** (not markdown). This is what a future versioning module's search/discovery tools will query ("which version of XenoCanto has a `32khz_path` column?"). CI regenerates it on every dataset change, so it's always in sync with the actual data.
>
> The existing `docs/dataset_schemas.md` markdown is a nice human-readable view but not sufficient for programmatic querying. The CI artifact should be JSON keyed by dataset name and version.

> **Discuss:** Which option? Is the current `docs/dataset_schemas.md` auto-generated approach sufficient for the "full" view?

## Question 3: Shared schema fragments

Many datasets share common column patterns:

**Taxonomy columns** (13 datasets):
`kingdom`, `phylum`, `class`, `order`, `family`, `genus`, `species_scientific`, `species_common`, `taxonomic_name`

**Detection/strong-label columns** (14 datasets):
`audio_path` (or `audio_fp`), `selection_table` (or `selection_table_fp`)

**Approach A: Composition via Python lists**

```python
TAXONOMY_COLUMNS = [
    ColumnSchema(name="kingdom", dtype="str"),
    ColumnSchema(name="phylum", dtype="str"),
    ColumnSchema(name="class", dtype="str"),
    ColumnSchema(name="order", dtype="str"),
    ColumnSchema(name="family", dtype="str"),
    ColumnSchema(name="genus", dtype="str"),
    ColumnSchema(name="species_scientific", dtype="str"),
    ColumnSchema(name="species_common", dtype="str"),
    ColumnSchema(name="taxonomic_name", dtype="str"),
]

DETECTION_COLUMNS = [
    ColumnSchema(name="audio_path", dtype="str"),
    ColumnSchema(name="selection_table", dtype="str"),
]

# Usage in a dataset:
schema = DatasetSchema(
    columns=[
        *TAXONOMY_COLUMNS,
        *DETECTION_COLUMNS,
        ColumnSchema(name="xeno_canto_id", dtype="str"),
    ]
)
```

**Approach B: Schema inheritance/merging**

```python
TaxonomySchema = DatasetSchema(columns=TAXONOMY_COLUMNS)
DetectionSchema = DatasetSchema(columns=DETECTION_COLUMNS)

schema = TaxonomySchema.merge(DetectionSchema).extend([
    ColumnSchema(name="xeno_canto_id", dtype="str"),
])
```

> **Author's position:** Not needed yet. Contract schemas are small (a handful of key columns per dataset), so shared fragments are premature abstraction. If we find ourselves copy-pasting 10+ column definitions across many datasets, we can revisit with simple list composition (Approach A). No new API needed now.

> **Discuss:** Is composition (Approach A) sufficient, or do we need formal schema merging? Where should shared column definitions live ‚Äî `schema.py` or a new `schema_fragments.py`?

## Question 4: Naming inconsistencies

Looking at the actual data, similar concepts have different column names across datasets:

| Concept | Variations |
|---------|-----------|
| Audio file path | `local_path`, `path`, `audio_path`, `audio_fp`, `gcs_path` |
| Audio file name | `file_name`, `audio_file_name`, `fn`, `Begin File` |
| Start time | `start`, `start_time`, `start_times(sec)`, `Begin Time (s)`, `vocal_onset` |
| End time | `end`, `end_times(sec)`, `End Time (s)`, `vocal_offset` |
| Species (scientific) | `species_scientific`, `scientific_name_unified`, `scientificName` |
| Species (common) | `species_common`, `vernacularName` |

**This is a separate problem from schemas**, but schemas force us to confront it.

**Options:**
1. **Accept it** ‚Äî each dataset has its own naming, schemas just document what exists
2. **Canonical aliases** ‚Äî schemas define a `canonical_name` for each column, datasets implement renaming
3. **Standardize incrementally** ‚Äî as we add schemas, normalize names where possible (breaking change per dataset)

> **Author's position:** Document as-is ‚Äî no breaking changes. But we should guide toward standardization over time:
> - Contract schemas use the **actual** column names in the data today.
> - We publish a **naming convention guide** (e.g., "for new datasets, prefer `audio_path` over `local_path`, `start_time` over `Begin Time (s)`").
> - New datasets follow the convention from day one.
> - Existing datasets migrate to standard names **only when they get a version bump** ‚Äî that's already a breaking change window, so renaming columns there is natural.
>
> This avoids the maintenance cost of canonical aliases while still driving convergence organically through the versioning lifecycle.

> **Discuss:** Should schemas address naming consistency, or is that a future concern? If we want canonical names, should that be a schema feature or an `output_take_and_give` convention?

## Question 5: Validation lifecycle

Currently `_validate_schema()` runs once after `_load()`. Other potential validation points:

| When | Why |
|------|-----|
| After `_load()` | Catch data file changes (current behavior) |
| After transforms | Ensure transforms didn't drop required columns |
| After `ConcatenatedDataset` merge | Ensure merged result has expected columns |
| On `__getitem__` | Validate individual rows (expensive, probably not) |

**Proposed answer:** Validate after `_load()` (current) and optionally after transforms. Concat validation should use the concat config's merge strategy rather than individual schemas.

> **Author's position:** After load only. Keep it simple ‚Äî transforms and concat are the user's responsibility. If a transform drops a required column, the user will discover that when they try to use it. Adding more validation points adds complexity for marginal benefit at this stage.

> **Discuss:** Are there other validation points we need? Should validation be opt-in (current: only if `schema` is defined) or required for all datasets?

## Question 6: Schema and dataset versioning

The versioning design doc (`Dataset_versioning_design.md`) proposes versioned datasets. Schema ties into this:

- When data changes (new version), the schema may change
- Should schemas be versioned alongside datasets?
- Should a version bump be required when schema changes?

**Proposed approach:** Schema is part of the dataset class definition. When a dataset version changes (e.g., AudioSet v0.1.0 -> v0.2.0 which adds `file_existed`, `32khz_path`, `16khz_path`, `audiosetcaps_caption`), the schema for that version reflects the new columns. The version system handles the rest.

> **Author's position:** Defer to the versioning design. The schema system shouldn't pre-decide how schemas attach to versions ‚Äî that's the versioning module's job. For now, the contract schema lives on the class, and the exhaustive CI-generated JSON is keyed by version. The versioning design will determine whether we need schema-per-version as a first-class concept.

> **Discuss:** Is tying schema to the class definition sufficient, or do we need schema-per-version as a first-class concept?

## Question 7: The `DType` system

Current `DType` literal:
```python
DType = Literal[
    "str", "int", "float", "bool", "datetime",
    "list[str]", "list[int]", "list[float]",
    "nullable",
]
```

**Open issues:**
- `"nullable"` is not a type ‚Äî it's a modifier. What does it mean as a standalone dtype?
- Should we support `"nullable[str]"`, `"nullable[int]"` etc.?
- Some columns are `float` only because they contain nulls (e.g., `gbifID` is `float` in some datasets because pandas promotes `int` + `null` to `float`). With polars, this isn't an issue. How should schemas handle this?
- `latitudeDecimal`/`longitudeDecimal` are `str` in some datasets but semantically should be `float` ‚Äî should schemas reflect actual storage dtype or intended dtype?

**Proposed answer:** Remove `"nullable"` as a standalone type. Schemas declare the intended dtype. Validation should coerce or warn (not fail) when the actual dtype is a known promotion (e.g., `int` -> `float` due to nulls).

> **Author's position:** Open question ‚Äî no strong recommendation yet. One concrete change: rename `"nullable"` to `"null"` for clarity. Beyond that, the right answer depends on how strict we want dtype validation to be, and whether schemas should reflect storage reality or semantic intent. This needs more discussion.

> **Discuss:** Should schemas declare actual dtypes or intended dtypes? How strict should dtype validation be?
